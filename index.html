Foundation 01

How the CPU Executes Instructions

The CPU runs exactly one process at a time. Scheduling is the decision of which one.

THE ONE RULE

A CPU can run exactly one process at a time per core. It switches between processes so fast it appears simultaneous. That switching is scheduling.

WHAT IT DOES

The CPU runs a simple loop forever, billions of times per second:

① FETCH → ② DECODE → ③ EXECUTE

Fetch — The CPU grabs the next instruction from memory using the Program Counter.

Decode — The control unit interprets what that instruction means (add, jump, load…).

Execute — The ALU performs the operation and stores the result in a register.

CPU: Live Execution Model
CONTEXT SWITCH

When the OS switches processes, it must save the current process's registers, program counter, and state — then load the next process's saved state. The CPU has no memory between runs. This save/restore is a context switch.

Foundation 02

Program → Process

A program on disk does nothing. The moment the OS runs it — it becomes a process.

The Transformation — Watch it happen
PROGRAM

A static file on disk — .exe, .out, a Python script. Just stored bytes. Does nothing until the OS loads it.

PROCESS

A running instance of a program. It has memory, a program counter, register values, and a state. Three Chrome windows = one program, three independent processes.

Foundation 03

CPU Burst vs I/O Burst

Processes don't use the CPU non-stop. They alternate — and this gap is where scheduling matters.

CPU BURST
  • → The process is actively using the CPU — calculations, logic, data processing
  • → This is when it appears on a Gantt chart
  • → Every process has at least one CPU burst
I/O BURST
  • → Process is waiting for something external — disk, network, keyboard
  • → CPU is completely free during this time
  • → Without scheduling, that free CPU sits idle — wasted
CPU ↔ I/O Alternation — Live Loop
Speed:
While P1 waits for I/O, the CPU runs P2. No wasted cycles. This is the whole point of scheduling.
CPU-BOUND

Long bursts, rare I/O. Examples: video encoding, ML training, scientific simulation.

I/O-BOUND

Short bursts, frequent waits. Examples: web servers, databases, file editors.

What is CPU Scheduling?

CPU Scheduling is the OS policy that decides
which process in the Ready Queue gets the CPU next — and for how long.

This single decision, made thousands of times per second, determines how fast your system feels, how fairly resources are shared, and whether any process gets starved.

Classification Axis — Preemptive vs Non-Preemptive
NON-PREEMPTIVE

Once a process starts running, it keeps the CPU until it voluntarily gives it up — either by finishing or by requesting I/O. The scheduler cannot interrupt it.

FCFS SJF Priority NP
PREEMPTIVE

The OS can forcibly remove a process from the CPU at any time — when a higher-priority process arrives, or when a time quantum expires.

SRTF Priority P Round Robin
LONG-TERM SCHEDULER

Admits programs into the system. Controls degree of multiprogramming. Runs rarely.

SHORT-TERM SCHEDULER ★

The one we study. Runs every millisecond — picks the next process from the ready queue and dispatches it to the CPU.

MEDIUM-TERM SCHEDULER

Swaps processes out of memory when RAM is full. Reloads them later. Manages degree of multiprogramming dynamically.

Criteria vs Algorithm: A scheduling criterion is what you optimise for (minimise waiting time, maximise throughput). A scheduling algorithm is the mechanism that achieves a criterion. SJF isn't just "an algorithm" — it's the direct response to the criterion of minimising average waiting time.

Stage 02

Key Metrics

Every algorithm is judged by these numbers. Know them cold.

ARRIVAL TIME (AT)
—

When the process enters the ready queue.

BURST TIME (BT)
—

Total CPU time the process needs to finish.

COMPLETION TIME (CT)
—

When the process finishes execution.

TURNAROUND TIME (TAT)
TAT = CT − AT

Total time in system — arrival to finish.

WAITING TIME (WT)
WT = TAT − BT

Time spent in the ready queue — not executing.

RESPONSE TIME (RT)
RT = First CPU start − AT

Time until the process first gets the CPU.

ADVANTAGES
  • → Simplest to implement — just a FIFO queue.
  • → No starvation — every process eventually runs.
  • → Low overhead — no sorting or priority checks needed.
DISADVANTAGES
  • → Convoy Effect — one long process blocks all short ones behind it.
  • → Poor average waiting time with mixed burst lengths.
  • → Not suitable for interactive or time-sensitive systems.
APPLICATIONS
  • → Simple batch processing systems.
  • → Print spoolers and job queues with similar burst times.
  • → Baseline — all other algorithms compared against FCFS performance.

Starvation — When Short Jobs Keep Arriving

P_long arrived at t=0 with BT=20. Short processes keep arriving. Watch P_long never get scheduled.

Starvation: P_long has been waiting while P2, P3, P4 all ran. If short jobs keep arriving, P_long may never run. Fix: Aging — gradually increase a waiting process's priority over time.
The Burst Prediction Problem: SJF/SRTF require knowing burst times in advance — but in reality, you can't know how long a process will run. Operating systems estimate burst time using exponential averaging of past behavior: τ(n+1) = α·t(n) + (1−α)·τ(n) where t(n) is the actual last burst and τ(n) is the estimate. This is why SJF is theoretically optimal but rarely used in pure form — real schedulers approximate it.
ADVANTAGES
  • → Minimum average waiting time of all non-preemptive algorithms.
  • → SRTF is the globally optimal preemptive algorithm — no other achieves lower avg WT.
  • → Short processes are served quickly — great turnaround for interactive-style jobs.
DISADVANTAGES
  • → Starvation — long processes may never run if short jobs keep arriving.
  • → Burst time must be known in advance — impossible in general-purpose systems.
  • → SRTF causes high context-switching overhead due to frequent preemptions.
APPLICATIONS
  • → Batch systems with known or predictable execution times.
  • → Systems that estimate burst time via exponential averaging of past bursts.
  • → Theoretical benchmark — every other algorithm's avg WT is measured against SRTF.
Algorithm 03

Priority Scheduling

Preemptive + Non-Preemptive · Criterion: Process Importance
THE IDEA

Each process has a priority number. CPU always picks highest priority. Convention: lower number = higher priority (P=1 beats P=4).

PREEMPTIVE VS NON-PREEMPTIVE

Non-preemptive: finishes even if higher priority arrives.

Preemptive: higher-priority arrival immediately kicks out current process.

SJF is Priority Scheduling where priority = 1/burst_time. This unifies the two algorithms under one framework.

ADVANTAGES
  • → Critical tasks guaranteed to run first — latency-sensitive work is never stuck behind low-priority jobs.
  • → Flexible — priority can be assigned based on any criterion (deadline, importance, resource class).
  • → Widely used in real-time OS and kernel schedulers where task ordering is mandatory.
DISADVANTAGES
  • → Starvation — low-priority processes may never run if high-priority work keeps arriving.
  • → Fix is aging — gradually increase a waiting process's priority over time.
  • → Deciding the right priority values for all processes is non-trivial in real systems.
APPLICATIONS
  • → Real-time systems — medical devices, aircraft control, industrial automation.
  • → OS kernel threads preempting user-space processes.
  • → Network packet scheduling — high-priority traffic (VoIP, video) routed first.
Algorithm 04

Round Robin

Preemptive · Criterion: Fairness · Time Quantum = Q

Every process gets a fixed time slice Q. After Q units it's preempted and goes to the back of the queue. Designed for fairness — no process waits forever.

Queue Rotation — How Round Robin Works
After P1's quantum expires, it moves to the back of the queue. P2 runs next. The queue rotates continuously.
The Quantum Dilemma: Q too large → behaves like FCFS. Q too small → context switches dominate, throughput collapses. The right Q is the key engineering decision.

ADVANTAGES
  • → No starvation — every process is guaranteed CPU time within one full rotation.
  • → Best response time for interactive systems — users get feedback quickly.
  • → Fair by design — all processes treated equally regardless of burst time.
DISADVANTAGES
  • → Higher average waiting time than SJF for the same process set.
  • → If Q is too small, context-switching overhead dominates and throughput collapses.
  • → Performance is highly sensitive to the choice of quantum Q — no one-size-fits-all value.
APPLICATIONS
  • → Time-sharing OS — the foundational model for modern multitasking.
  • → Interactive desktops and terminals where responsiveness is critical.
  • → Base layer of multi-level feedback queue schedulers used in Linux and Windows.
